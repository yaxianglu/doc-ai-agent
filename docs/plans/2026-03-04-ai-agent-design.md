# AI 智能体（指挥调度）设计文档

## 1. 目标
- 重点能力：基于平台真实数据进行可核验问答（统计、TopN、按时间筛选）。
- 亮点能力：结合外部知识给出处置建议，并标注来源与时间。
- 实时口径：数据到达后 15 分钟内可被查询，问答响应 10 秒内。

## 2. 约束与现状
- 当前数据源仅为导出报表（xlsx）。
- 项目当前为空仓库，无既有服务框架。
- 需先保证“真实数据可追溯”再扩展联网建议。

## 3. 方案对比
### 方案 A：纯 RAG
- 优点：开发快。
- 缺点：统计口径不稳定，难满足“结果必须来自真实数据并可复算”。

### 方案 B：纯 Text2SQL
- 优点：统计类问题可核验。
- 缺点：建议类能力较弱，无法直接提供外部来源处置建议。

### 方案 C：混合路由（推荐）
- 统计类问题走结构化查询（SQL）。
- 建议类问题走建议引擎（先本地规则，预留联网检索接口）。
- 兼顾需求 1 的准确性和需求 2 的扩展性。

## 4. 架构设计
- 数据接入层：解析 xlsx，标准化字段，写入 SQLite（MVP）并支持增量刷新。
- 查询层：
  - 意图识别（统计查询 / 建议查询）。
  - 统计查询：自然语言映射为受控 SQL 模板。
  - 输出：答案 + 过滤条件 + 统计 SQL + 时间窗。
- 建议层：
  - MVP 使用规则建议（基于告警类型和等级）。
  - 预留 WebProvider 接口用于后续联网检索。
- 服务层：HTTP API（`/chat`、`/refresh`、`/health`）。

## 5. 数据模型（MVP）
核心表 `alerts`：
- `id`（自增）
- `alert_content`
- `alert_type`
- `alert_subtype`
- `alert_time`
- `alert_level`
- `region_code`
- `region_name`
- `alert_value`
- `device_code`
- `device_name`
- `longitude`
- `latitude`
- `city`
- `county`
- `sms_content`
- `disposal_suggestion`
- `source_file`
- `source_sheet`
- `source_row`

## 6. 关键流程
1. 定时扫描数据目录（默认每 5 分钟）。
2. 对新增/更新 xlsx 执行解析并写入数据库。
3. `/chat` 接收问题：
   - 若为统计查询，执行 SQL 并返回证据。
   - 若为建议查询，返回建议与适用范围。
4. 所有结果保留可追溯信息。

## 7. 错误与安全
- SQL 仅允许模板化查询，禁止任意 SQL 执行。
- 缺字段或时间解析失败时记录告警并跳过坏行。
- 空结果明确返回“无匹配数据”，避免幻觉描述。

## 8. 测试策略
- xlsx 解析测试（表头映射、行解析）。
- 仓储测试（写入与聚合查询）。
- 查询引擎测试（“多少”“top5”）。
- API 集成测试（`/chat` 和 `/refresh`）。

## 9. 里程碑
- M1（本次）: 可运行 MVP，覆盖需求1主链路 + 需求2占位能力。
- M2（后续）: 接入实时事件流、联网检索、鉴权与审计。
