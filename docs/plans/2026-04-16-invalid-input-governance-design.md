# Invalid Input Governance Design

## Goal

在不重写现有 Agent 主架构的前提下，为系统补一条统一、可测试、可扩展的“无效输入治理链路”，解决以下问题：

1. 乱码、键盘乱敲、碎片输入会被错误识别成业务问题。
2. `router_advice` 这类模型路由结果缺少执行前闸门，容易把低质量输入放进建议链路。
3. 系统当前更关注“怎么回答”，但对“这条输入是否值得回答”判断不足。
4. 输出层缺少针对“输入不成立却答得很像样”的最终拦截。

本轮希望把“无效输入处理能力”从当前约 `2.5/10` 提升到第一阶段目标 `7/10`，后续稳定目标 `8.5/10`。

## Scope

本轮只做高收益、低风险的治理补强：

- 增加统一的 `Input Guard`，在业务规划之前先判断输入是否具备可理解性。
- 给 `Intent Router` 增加执行前校验，不允许仅凭模型猜测直接进入 advice/query 链路。
- 为 advice/data_query 增加统一 `Execution Gate`，要求满足最小执行条件。
- 在 `Answer Guard` 增加“输入不成立却生成业务结论”的兜底重写/回退。
- 增加一组“无效输入专项回归测试”，覆盖单轮、多轮、越界、会话污染等场景。

不做的内容：

- 不引入新的外部模型或 moderation 服务作为主判定链路。
- 不重写 LangGraph 工作流。
- 不把所有自然语言理解逻辑迁移到新模块。
- 不做大规模知识库策略改造。

## Current Problem

当前线上链路的实际问题不是“文案不够好”，而是“输入治理不足”：

1. `h d k j h sa d k l j` 这类输入，在真实线上配置下可被判成 `advice`。
2. 一旦进入 advice 路径，系统会继续执行检索 + LLM 生成，于是产出貌似合理但实际上无依据的农业建议。
3. 这类错误比“回答啰嗦”更严重，因为它会让系统对无意义输入给出业务动作建议。

本质问题是：**系统缺少“先判断能不能回答，再决定回答什么”的分层治理。**

## Design

### 1. Input Guard

新增统一输入治理入口，建议落在理解/规划前的轻量规则层，优先负责“是否像有效问题”判断。

建议输出统一结果：

- `is_valid_input: bool`
- `invalid_reason: str`
- `should_clarify: bool`
- `clarification: str | None`
- `confidence: float`

第一阶段重点拦截：

- 键盘乱敲：`h d k j h sa d k l j`
- 英文/拼音碎片：`asd qwe zxc`
- 纯符号/数字噪声
- 超低信息输入：仅 1-2 个无语义 token
- 非自然语言形式的无结构碎片

这里不追求“理解所有输入”，而是优先识别**明显不值得进入业务链路**的输入。

### 2. Domain Guard

在输入通过 `Input Guard` 后，再做能力边界判断。

目的：

- 区分“无效输入”和“有效但越界输入”。
- 对越界输入继续走现有 out-of-scope 能力回复。
- 对无效输入统一走澄清，不进入业务规划。

例如：

- `浙江天气` -> 越界
- `h d k j h sa d k l j` -> 无效输入

这两类都不应该进入业务建议/数据查询链路，但回复策略不同。

### 3. Router as Candidate, Not Authority

保留现有 `Intent Router`，但它只能提供候选结果，不能直接拥有放行权。

核心改动：

- `router_advice` 不再天然可信。
- 模型若给出 `advice`，仍需通过显式 advice 信号校验。
- 模型若给出 `data_query`，仍需通过最小业务槽位校验。

这意味着：

- 路由器负责“猜测意图”
- 闸门负责“决定是否执行”

### 4. Execution Gate

在 `QueryPlanner` 形成最终 plan 前增加统一执行前校验。

#### 4.1 Advice Gate

只有满足以下任一条件，才允许进入 advice 链路：

- 原始问题中存在显式建议词：
  - `建议`
  - `怎么办`
  - `如何`
  - `怎么处理`
  - `处置`
  - `防治`
- 或者该输入被识别为高置信 advice follow-up，且具备明确上下文依托。

否则，即使 router 返回 `advice`，也必须降级为：

- `clarification`
- 或 `invalid_input_clarification`

#### 4.2 Data Query Gate

进入数据查询链路前，至少满足其中之一：

- 有明确领域（虫情/墒情/预警）
- 有明确地区、设备或时间范围
- 有明确趋势/排行/明细等数据意图词

对于完全无槽位、无领域、无结构的问题，不允许进入 `data_query`。

### 5. Answer Guard

输出前增加一层输入一致性守卫，专门拦截这类情况：

- 输入本身不成立
- 但最终回答输出了农业建议、虫情概况、墒情分析、处置动作

当命中时：

- 优先改写为统一澄清回复
- 不保留原业务答案

这层用于兜住“前面漏网，但输出已经开始编业务内容”的情况。

### 6. Unified Clarification Strategy

不同失败类型采用统一、短句、可引导的澄清回复。

#### 无效输入

`我没看懂这条输入。你可以直接问虫情、墒情、预警数据，或让我给处置建议。`

#### 领域不明

`你想看虫情还是墒情？`

#### 对象不明

`请补充地区、设备或时间范围，我再继续。`

目标是让系统在失败时也稳定、简洁、可继续对话，而不是随意输出业务内容。

## Testing Strategy

新增一组“无效输入治理专项回归”：

1. **无效输入**
   - 键盘乱敲
   - 英文碎片
   - 纯符号
2. **模糊输入**
   - 半截句
   - 无对象代词
3. **越界输入**
   - 天气
   - 新闻
   - 购票
4. **多轮风险**
   - 上一轮是虫情/墒情，下一轮突然乱输
   - 上一轮是建议追问，下一轮突然乱输
5. **误伤控制**
   - 确保真实问题不会被大量误判成无效输入

专项测试的重点不是“答得多漂亮”，而是：

- 是否正确拒答/澄清
- 是否阻断 advice/query 执行链路
- 是否避免业务幻觉

## Metrics and Scoring

建议把“无效输入治理能力”单独评分，不混入总问答分。

### Current Score

- 当前能力分：`2.5/10`

原因：

- 单轮乱码会误入 `advice`
- 最终还能产出业务建议
- 缺少前置治理与输出拦截

### Phase 1 Target

- 目标分：`7/10`

达标标准：

- 绝大多数乱码、低信号输入都转为澄清
- 不再出现“乱码 -> 农业建议/农业数据概况”

### Phase 2 Target

- 目标分：`8.5/10`

达标标准：

- 同时稳住单轮、多轮、越界、模糊输入
- 真问题误伤率保持在可接受范围

### Suggested Operational Metrics

- 无效输入误答率 `< 1%`
- 无效输入澄清率 `> 95%`
- 无效输入进入 advice/query 执行链路 `< 1%`
- 真问题误伤率 `< 5%`

## Risks and Mitigations

- 风险：规则写太死，误伤真实问题。
  - 规避：先覆盖明显无效输入；保留保守阈值；配套误伤测试。
- 风险：路由和守卫重复判断，逻辑变绕。
  - 规避：明确职责边界：router 负责猜测，gate 负责放行。
- 风险：多轮上下文把无效输入重新包装成 follow-up。
  - 规避：无效输入优先级高于 follow-up 复用。
- 风险：输出守卫过晚，成本浪费。
  - 规避：输出守卫只做最后兜底，主治理放在前置闸门。

## Success Criteria

- `h d k j h sa d k l j` 这类输入稳定返回澄清，不进入业务建议链路。
- `router_advice` 不再能覆盖明显无效输入。
- advice/query 只有在满足最小条件时才执行。
- 多轮上下文不再把乱码自动继承成业务 follow-up。
- 新增专项回归测试后，可持续评估该能力并追踪分数提升。
